# wxhook 重构实施计划

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** 在不破坏 Xposed Hook、微信数据库处理、root 备份和云同步能力的前提下，将 wxhook 从“UI 直接编排 shell/文件操作”重构为可验证的分层架构。

**Architecture:** 采用分层 + 用例编排架构：UI/Receiver 只发起用例；Service/Worker 承载长任务；UseCase 编排流程；RootGateway 统一受保护路径访问；BackupRepository 管理备份包；CloudClient 管理 WebDAV/rclone；DB 层只负责数据库和解析。先统一边界和返回值，再迁移调用点，最后拆分大文件与 UI。

**Tech Stack:** Kotlin/JVM 17、Android API 35、Coroutines、Lifecycle/ViewModel、现有 Room（仅用于 wxhook 自己的消息缓存/界面数据，不承载备份文件状态）、现有 `RootCommandRunner`、tar/zstd/sqlcipher、rclone；WebDAV 独立适配 `sardine-next`。真实验证以 Android 13 rooted device、`pm install -r`、root shell、Xposed/LSPosed 和 GitHub Actions 构建为准。

### 现状风险

1. `Runtime.exec`, `su`, `File("/sdcard/...")`, `File("/data/...")`、rclone 和 shell 字符串仍散落在多个层。
2. `RootCommandRunner` 已存在，但只统一了部分命令；复杂脚本仍自行拼接和吞错。
3. `SyncService` 启动 rclone 后没有严格依据 exit code、超时和 stderr 判定成功。
4. `SetupManager` root 拷贝失败时仍可能写 `.setup_done`。
5. `BackupPackage`、`BackupHookLocal` 中存在路径拼接、权限设置和命令结果判定风险。
6. `/sdcard` 普通 App 访问、root 访问、临时 `/data/local/tmp` 访问没有统一抽象。
7. `AndroidManifest.xml` 中 `SettingsActivity` 重复声明；权限声明与实际 AppOps 状态没有统一检查。
8. `ModuleActivity` 约 476 行、`MessageParser` 约 305 行，UI、编排和数据处理边界混杂。
9. 目前没有发现稳定的单元测试目录；不能用 JVM 模拟替代最终真实设备验证。
10. 当前工作区有未提交改动：约 `32` 个文件新增、`20` 个文件删除/重命名状态。重构前必须保存并审计现状，不覆盖用户改动。

### 不可违反的约束

- 不卸载 App；设备验证使用 `pm install -r`。
- 最终用户可见备份直接位于 `/sdcard/Download/wxhook_backup/`。
- 临时文件可用 `/data/local/tmp`，但不把临时路径当最终备份路径。
- 访问微信 `/data` 使用 root 数据平面；sdcard 权限不作为 `/data/data` 权限替代。
- 备份前停止微信；数据库备份必须处理 WAL/journal 一致性。
- 不用 `chmod -R 644`；目录和文件权限分开处理。
- 每个阶段完成后都要构建并记录真实输出；专项脚本只能补充，不能替代真实 App/设备验证。
- 不在重构中顺带升级无关依赖、修改签名、改变 applicationId 或删除旧功能。

## 1A. 数据边界：微信原始数据、备份文件、wxhook 自身状态必须分离

### A. 微信原始数据：只读备份源，不归 wxhook 所有

```text
/data/user/<userId>/com.tencent.mm/
/data/user_de/<userId>/com.tencent.mm/
/proc/<wechatPid>/root/...
```

这里的 `EnMicroMsg.db`、WAL/journal、`image2/voice2/video/emoji/avatar` 等是微信原始数据。wxhook 只负责：

```text
root 定位
停止微信
只读复制/快照
读取 SQLCipher
读取文件 metadata
```

禁止：

```text
把微信原始消息/联系人/会话重新写入 wxhook.db 作为备份主数据
修改微信原始目录中的文件
用 Room 代替微信数据库
把微信原始 DB 放进 App 私有数据库
```

Room 里的 `CachedMessage`、`CachedContact`、`CachedConversation` 若继续保留，只能定义为 wxhook UI 查询缓存/导入结果，不是备份源，也不是备份文件索引；需要标明 `source` 和 `snapshotTime`，不能把缓存当作完整微信数据。

### B. 备份数据：用户可见归档，使用文件 + JSON

最终目录固定：

```text
/sdcard/Download/wxhook_backup/
```

保存：

```text
archives/full_*.tar.gz 或 full_*.tar.zst
archives/incr_*.tar.gz 或 incr_*.tar.zst
manifest.json
backup_records.json
backup_live.log
remote_config.json
```

这些不是微信原始数据库，也不放入 `wxhook.db`。它们必须能被 USB、Termux、root shell 和未来版本直接读取。

### C. wxhook 自身状态：App 私有文件/SharedPreferences

只保存运行状态和 UI 偏好：

```text
/data/data/com.nous.wxhook/shared_prefs/
/data/data/com.nous.wxhook/files/
```

例如：

```text
首次启动状态
UI 主题/开关
最后打开页面
任务取消标志
日志级别
```

不要把最终备份记录只放这里；卸载/清理 App 后，备份历史必须仍可从 `/sdcard/Download/wxhook_backup/` 恢复。

### 存储决策

```text
微信原始数据       → 不保存副本到 Room；复制为归档文件
备份归档本体       → tar.gz/tar.zst
备份 manifest       → JSON
备份历史记录       → JSON
云端配置           → JSON（密码/token 不明文放共享目录）
wxhook UI 缓存      → Room，可选
wxhook UI 偏好      → SharedPreferences/DataStore
密钥               → App 私有安全存储，不放 manifest/共享目录/log
```

### 禁止混淆的命名

```text
wechatSourcePath       微信当前原始源路径
stagingPath            root 临时快照路径
archivePath            最终备份归档
backupManifestPath     备份文件清单
wxhookCacheDatabase    wxhook UI 缓存数据库
```

`manifest.json` 记录的是“已进入备份系统的文件”，不是微信数据库内容索引；`wxhook.db` 记录的是“wxhook 为界面查询缓存的内容”，不是备份归档状态。

## 2. 目标架构

### 2.1 包结构

在保持现有包兼容的前提下，逐步迁移到：

```text
com.nous.wxhook/
├── app/                Application、启动初始化、能力探测
├── core/model/         Result、错误、路径、备份清单、任务状态
├── core/command/       ShellCommand、CommandResult、超时/日志/转义
├── root/               RootGateway、RootService/RootCommand 实现
├── storage/            公共存储、临时目录、私有配置、权限/AppOps
├── backup/             BackupUseCase、BackupOrchestrator、ArchiveService
├── sync/               CloudSyncUseCase、CloudClient、RcloneClient、WebDavClient
├── database/           DbUseCase、SQLCipher、导出/合并/解析
├── worker/             Foreground/WorkManager 或现有 Service 适配
├── receiver/           仅事件转发，不做业务
└── ui/                 Activity、ViewModel、UI state/event
```

第一阶段不强制移动所有文件；先建立接口和统一返回值，迁移稳定后再用 `git mv` 移动文件。

### 2.2 依赖方向

```text
ui/receiver
    ↓
worker/service
    ↓
usecase/orchestrator
    ↓
root/storage/database/sync interfaces
    ↓
platform implementations
```

禁止：

```text
UI → Runtime.exec
UI → RootCommandRunner
UI → /data 或 /sdcard File 直接读写
DB parser → Activity/Service
Cloud client → UI 控件
```

### 2.3 统一结果模型

新增 `core/model/OperationResult.kt`：

```kotlin
sealed interface OperationResult<out T> {
    data class Success<T>(val value: T, val warnings: List<String> = emptyList()) : OperationResult<T>
    data class Failure(val error: WxHookError, val warnings: List<String> = emptyList()) : OperationResult<Nothing>
}

enum class WxHookError {
    ROOT_UNAVAILABLE,
    COMMAND_FAILED,
    COMMAND_TIMEOUT,
    PATH_NOT_FOUND,
    STORAGE_ACCESS_DENIED,
    INVALID_CONFIGURATION,
    TARGET_STILL_RUNNING,
    ARCHIVE_INVALID,
    METADATA_RESTORE_FAILED,
    CLOUD_AUTH_FAILED,
    CLOUD_UPLOAD_FAILED,
    DATABASE_FAILED,
    CANCELLED,
    UNKNOWN,
}
```

所有 root、打包、解密、rclone/WebDAV、服务完成广播都使用结构化结果；UI 只渲染结果，不自行猜测成功。

## Phase 1：统一命令执行和路径安全边界

目标：不移动业务文件，先让所有受保护操作有一个可检查入口。

### Task 1.1：扩展 `RootCommandRunner`

**Files:**
- Modify: `app/src/main/java/com/nous/wxhook/rootbridge/RootCommandRunner.kt`
- Create: `app/src/main/java/com/nous/wxhook/core/command/CommandResult.kt`
- Create: `app/src/main/java/com/nous/wxhook/core/command/ShellEscaper.kt`

**Design:**

```kotlin
data class CommandResult(
    val exitCode: Int,
    val stdout: String,
    val stderr: String,
    val timedOut: Boolean,
) {
    val isSuccess: Boolean get() = !timedOut && exitCode == 0
}
```

要求：

- 同时消费 stdout/stderr，避免子进程死锁。
- 超时必须 destroy/cleanup，并返回 `timedOut=true`。
- 保留 stderr，不再只返回“stdout 或 stderr”。
- root 命令开始时可选执行 `id -u` 验证 UID 0。
- 用户输入路径禁止直接字符串插入；提供单参数 shell quoting 或优先使用参数数组。
- 长命令输出写临时日志，结果只保留摘要和尾部错误。

**Verification:** 单元测试覆盖成功、非零退出、超时、引号路径、stderr；再运行 `:app:compileDebugKotlin`。

### Task 1.2：建立 `RootGateway`

**Files:**
- Create: `app/src/main/java/com/nous/wxhook/root/RootGateway.kt`
- Create: `app/src/main/java/com/nous/wxhook/root/RootGatewayImpl.kt`
- Modify: `RootCommandRunner.kt`

接口至少包含：

```kotlin
interface RootGateway {
    suspend fun check(): OperationResult<RootStatus>
    suspend fun exists(path: String): OperationResult<Boolean>

    suspend fun mkdirs(path: String): OperationResult<Unit>
    suspend fun copy(src: String, dst: String): OperationResult<Unit>
    suspend fun delete(path: String): OperationResult<Unit>

}
```

实现初期仍可内部调用现有 `RootCommandRunner`，不要第一阶段就切 libsu。这样可以先统一行为，后续再替换实现。

## Phase 3：重构备份核心流程

### Task 3.1：拆分 `BackupHookLocal`

```text
Orchestrator          按顺序编排，不执行 UI 操作
```

### Task 3.2：明确备份时序

```text
1. 检查 root UID 0 和 mount namespace。
2. 解析 userId，默认只作为 UI 默认值，不作为唯一事实。
3. force-stop com.tencent.mm。
4. 确认 pidof 不再返回目标进程。
5. 读取 DB/WAL/journal 和附件源路径。
6. 捕获 UID/GID/mode/SELinux context。
7. 打包到任务临时目录。
8. 用 tar -tf 或 zstd -d -c | tar -tf 验证归档。
9. 原子复制到 `/sdcard/Download/wxhook_backup/`。
10. 校验最终文件大小/存在性/manifest。
11. 记录成功或失败结果。
12. 按用户设置重启微信。
```

### Task 3.3：修复权限和归档错误

- 删除所有 `chmod -R 644`。
- 目录用 `755`，普通文件用 `644`，但不覆盖 Android data 的原始 metadata。
- `BackupPackage` 的 tar、cp、stat、rm 每一步检查 exit code。
- 不把小于某个字节数作为唯一有效性判断；用 tar 列表、manifest 和文件大小共同判断。
- shell 命令使用 `set -o pipefail`。
- 压缩格式与扩展名保持一致：gzip=`.tar.gz`，zstd=`.tar.zst`。

### Task 3.4：引入 manifest 版本

```json
{
  "schema": 1,
  "package": "com.tencent.mm",
  "userId": 0,
  "createdAt": 0,
  "compression": "zstd",
  "entries": [],
  "sourceMetadata": {}
}
```

恢复时拒绝未知 schema；向后兼容旧备份时建立显式迁移器，不在解析器中散落兼容分支。

## 8A. 微信密钥 Hook 专项方案

### 当前实现与问题

当前 `xposed/.../KeyCaptureHook.kt`：

```text
Application.attach → 扫描 com.tencent.wcdb.core.Database
→ hook setCipherKey 所有 declaredMethods
→ 写 /data/local/tmp/.wechat_key
→ ContentProvider 插入
→ 另有 KeyReceiver 广播链路
```

主要问题：

- `hookAllMethods(Application, "attach")` 范围过大，可能重复安装。
- `keyCaptured` 是进程级布尔值，无法处理多次调用、进程重建和 key 轮换。
- 不区分“Hook 安装成功”“发现 setCipherKey”“捕获 key”“key 可解密”。
- 完整 key 出现在 Xposed 日志、Logcat、全局临时文件和广播链路，泄漏面过大。
- 固定 `com.tencent.wcdb.core.Database` 和 `setCipherKey` 签名，微信版本变化后诊断信息不足。
- native `libwcdbhook.so` 当前是额外依赖，不应成为 Java Hook 启动的单点故障。

### 目标数据流

```text
微信主进程
  → 精确 attach(Context) Hook，仅安装一次
  → WCDB setCipherKey 精确 overload Hook
  → KeyCandidateValidator
  → KeyCaptureStore（内存 + 管理 App 私有存储）
  → SQLCipher 只读探测
  → 状态：INSTALLED / INVOKED / CAPTURED / VERIFIED / FAILED
```

### 实施任务

1. 新建 `CapturedKey`、`KeyCandidateValidator`、`KeyCaptureStore`。
2. 只 Hook 精确的 `Application.attach(Context)`，记录已处理 classloader 和 processName。
3. 只在微信主进程安装 WCDB Hook；不在 `:push` 等子进程重复安装。
4. 捕获 callback 只复制 byte array、读取 metadata、做格式校验和去重；不做网络、数据库、压缩或 UI 操作。
5. `setCipherKey` 的每个 overload 记录参数类型和签名；不假设 page size/version 的参数位置。
6. 校验空数组、全零、异常长度、非 byte-like 参数；允许实际版本差异，不把单一长度写死成唯一规则。
7. 普通日志只记录 key SHA-256 fingerprint、长度、微信版本、方法签名和时间，禁止打印完整 key。
8. 捕获后先执行 SQLCipher 只读探测；探测成功才标记 `VERIFIED`，不能因 Hook 回调发生就显示“密钥可用”。
9. 删除 `/data/local/tmp/.wechat_key` 明文旁路；key 写入管理 App `noBackupFilesDir/key/`，或通过受保护 Provider 一次性读取。
10. Provider 不再公开静态全局 key；跨进程读取必须有 signature 权限/受控 token。广播最多传 record ID、fingerprint 和状态，不传 key。
11. `KeyReceiver` 改为内部状态接收器或删除，Manifest 去除不必要的 `exported=true`。
12. native Hook 只作为明确版本适配后的 fallback；加载失败不得影响 Java Hook 和微信启动。

### 密钥 Hook 验收

```text
冷启动：Hook 仅安装一次
进程重建：可再次安装但不重复写入同一 fingerprint
主进程/子进程：只捕获目标进程
setCipherKey 多次调用：同 fingerprint 去重，换 key 可重新记录
微信升级/混淆变化：输出候选类和方法签名诊断
捕获 key 后 SQLCipher 只读探测成功
无管理 App/Provider：Hook 不崩溃，可稍后重试
logcat、Xposed log、sdcard：无完整 key
```

## 8B. Git 完全弃用：Manifest 对比 + 增量归档

### 目标

完全删除 Git。每次备份只比较上一次 manifest 与当前文件状态，记录新增/修改/删除，然后把本次新增/修改文件打包成一个归档。

```text
读取 last_manifest.json
→ 扫描当前源目录
→ 对比相对路径、文件大小、mtime
→ 对疑似变化文件计算 SHA-256
→ 得到 added / changed / deleted
→ added + changed → 本次增量归档
→ 校验归档
→ 保存本次 manifest 和 backup_records
```

`deleted` 只记录状态，不从历史归档删除。这样旧归档仍可用于恢复历史文件。

### 目录结构

```text
/sdcard/Download/wxhook_backup/
├── manifest.json                 当前文件清单
├── backup_records.json           每次备份记录
├── archives/
│   ├── full_20260713.tar.zst    首次全量归档
│   ├── incr_20260713_220000.tar.zst
│   └── ...
└── <user_hash>/                  可选：解包后的当前文件树
```

manifest 条目：

```json
{
  "path": "user_hash/image2/aa/bb.jpg",
  "size": 12345,
  "mtime": 0,
  "sha256": "...",
  "kind": "file"
}
```

backup record：

```json
{
  "id": "20260713_220000",
  "type": "incremental",
  "archive": "archives/incr_20260713_220000.tar.zst",
  "compression": "zstd",
  "added": 12,
  "changed": 3,
  "deleted": 1,
  "size": 123456,
  "durationMs": 0,
  "verified": true
}
```

### Kotlin 组件

```text
backup/incremental/FileManifest.kt
backup/incremental/ManifestStore.kt
backup/incremental/FileDiff.kt
backup/incremental/IncrementalArchiveService.kt
```

核心模型：

```kotlin
data class ManifestEntry(
    val relativePath: String,
    val size: Long,
    val mtime: Long,
    val sha256: String,
)

data class FileDiff(
    val added: List<ManifestEntry>,
    val changed: List<ManifestEntry>,
    val deleted: List<String>,
)
```

### 对比算法

1. 读取旧 manifest；文件不存在时视为空 manifest。
2. 使用 root `find/stat` 扫描源目录，不能依赖 App 进程对 `/data` 或 root-owned `/sdcard` 的 `File.listFiles()`。
3. 对每个当前文件生成相对路径、size、mtime。
4. 旧条目不存在 → `added`。
5. size 或 mtime 不同 → 计算当前 SHA-256；hash 不同 → `changed`；hash 相同 → 仅 metadata 变化，可按配置忽略或记录。
6. 旧条目在当前扫描中不存在 → `deleted`。
7. 首次运行必须生成全量归档；没有新增/修改时不创建空归档，只记录“无变化”。
8. 排序路径后再写 manifest，保证同一输入产生稳定内容，便于校验。

### 增量归档流程

```text
1. 停止微信并确认目标进程退出。
2. 将源文件复制/快照到 /data/local/tmp/wxhook/<taskId>/，避免直接在 FUSE 上打大包。
3. 生成当前 manifest 和 FileDiff。
4. added + changed 文件按相对路径写入临时归档目录。
5. 使用 tar + gzip/zstd 打包；扩展名必须匹配实际压缩算法。
6. 用 tar -tf 或 zstd -d -c | tar -tf 验证归档可读。
7. 复制归档到 `/sdcard/Download/wxhook_backup/archives/`。
8. stat 校验最终归档存在且大小大于 0。
9. 原子写入 `manifest.json` 和 `backup_records.json`。
10. 最后清理 `/data/local/tmp/wxhook/<taskId>/`。
```

中途失败时：

- 不覆盖旧 manifest。
- 不推进 `backup_state.json`。
- 不追加 `verified=true` 记录。
- 临时归档保留到错误清理阶段或记录路径，方便诊断。

### DB 文件处理

数据库增量 SQL 归档沿用现有 rowid 方案，但和附件文件 diff 分开记录：

```text
EnMicroMsg_baseline.sql.gz       首次全量
incr_<from>_to_<to>.sql.gz       后续消息增量
```

最终增量归档可包含：

```text
<user_hash>/incr_*.sql.gz
<user_hash>/新增附件...
<user_hash>/修改附件...
manifest.json
```

只有最终归档存在且非空，才更新 `lastMessageRowId`、manifest 和 backup record。

### Git 删除清单

从以下位置完全移除 Git：

```text
app/src/main/assets/bin/git
SetupManager.BINS
SetupManager.EXEC
SetupManager.git(ctx)
BackupHookLocal.gitAddAndCommit()
所有 git init/add/commit/rev-parse 调用
UI/README 中的 Git commit 文案
```

旧 `/sdcard/Download/wxhook_backup/.git` 不自动删除，避免误删历史；提供一次性“删除旧 Git 目录”操作前显示路径、大小，并要求用户主动执行。新版本不再读取或写入它。

### 验收

```text
首次备份：生成 full archive + manifest
第二次无变化：不生成空 archive，记录无变化
新增文件：只进入本次增量 archive
文件内容变化但 size/mtime 异常：hash 能识别
文件删除：记录 deleted，不修改旧 archive
gzip/zstd：扩展名、实际编码、解码命令一致
归档损坏/复制失败：不更新 manifest
进程中断：旧 manifest 仍可用
大文件：流式 hash，不 OOM
路径 ../ 或 symlink 越界：拒绝
```

## 8C. 联合验收原则

```text
Hook 安装成功 ≠ 捕获 key
捕获 key ≠ key 可解密
本地增量归档生成 ≠ 云端上传成功
归档存在 ≠ 归档完整可解压
manifest 更新成功 ≠ 归档已验证
```

UI、日志、记录文件分别显示这些状态，禁止合并成一个“成功”标志。
